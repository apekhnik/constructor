# Интерактивный конструктор электрощита — техническое задание и план реализации

## 1. Что мы строим

Веб-приложение (одна страница, HTML/CSS/JS, без бэкенда), в котором пользователь
собирает однолинейную/монтажную схему электрощита: перетаскивает компоненты на
DIN-рейку, соединяет их клеммы проводами, подаёт питание и наблюдает реальную
электрическую логику — что включится, что не включится, что сработает на
аварии и почему. Цель — обучение и тренаж: пользователь должен научиться
правильно собирать щит и понимать, почему неправильная сборка не работает или
опасна.

Это не SPICE-симулятор и не считает токи с точностью до ампера. Это
**логический симулятор уровня "сработает/не сработает"**, основанный на
реальных принципах работы аппаратов защиты. Точность должна быть достаточной,
чтобы объяснять физику, но без полноценного расчёта токов короткого замыкания,
падения напряжения в проводах и т.п.

Дизайн интерфейса (внешний вид, тема, расположение, визуальный язык) будет
сделан отдельно через Claude Design по отдельному промпту (см. файл
`design-prompt.md` в этой же папке). Этот документ описывает **логику,
данные и поведение**, а не то, как это должно выглядеть. План реализации
ниже учитывает, что готовый дизайн/вёрстку нужно будет интегрировать с этой
логикой, поэтому architecture специально разделяет "движок" (engine) и "UI
слой" (renderer), чтобы дизайн можно было подключить как слой представления
над готовой моделью данных.

## 2. Электротехническая модель

### 2.1 Общие принципы сети

- Однофазная сеть 220 В (на будущее можно расширить до 3-фазной 380 В, но в
  MVP — однофазная).
- Три типа проводников: **L** (фаза), **N** (рабочий ноль), **PE** (защитная
  земля). PE никогда не коммутируется аппаратами защиты — он просто
  физически соединяет корпуса и шину заземления.
- Источник питания: внешняя сеть с параметром "текущее напряжение сети"
  (по умолчанию 220 В, пользователь может двигать слайдер 0–300 В, чтобы
  тестировать реле напряжения), плюс кнопка "обрыв нуля" и кнопка "короткое
  замыкание на корпус" / "утечка тока" для тестирования УЗО.
- Граф схемы — это узлы (клеммы компонентов) и рёбра (провода). Симуляция —
  это распространение состояния "под напряжением / обесточено" и значения
  тока по графу от источника к нагрузкам, с учётом того, разомкнут ли каждый
  коммутирующий аппарат.

### 2.2 Модель компонента

Каждый компонент описывается как:

```js
{
  id: "uuid",
  type: "breaker" | "rcd" | "diff_breaker" | "voltage_relay" | "three_way_switch" | "contactor" | "bus_din" | "bus_pe" | "load" | "indicator_lamp" | "source",
  label: "Автомат C16",
  rated_current_A: 16,        // для автоматов
  curve: "B" | "C" | "D",     // время-токовая характеристика
  rated_leak_mA: 30,          // для УЗО / дифавтоматов
  poles: 1 | 2,               // однополюсный / двухполюсный
  terminals: [
    { id: "t1", role: "in_L" },
    { id: "t2", role: "out_L" },
    // ...
  ],
  state: {
    on: true,                 // тумблер включен/выключен пользователем
    tripped: false,           // отключился аварийно (нужен ручной ресет)
    trip_reason: null,        // "overload" | "short_circuit" | "leak" | "overvoltage" | "undervoltage" | "no_neutral"
  },
  position: { rail: 1, slot: 4 } // позиция на DIN-рейке
}
```

### 2.3 Компоненты для MVP (минимальный набор)

Список ниже — то, что должно быть реализовано в первой версии. Это
осознанно небольшой набор, достаточный, чтобы собрать реалистичную и
показательную схему ввода + 1-2 отходящие линии + резервный источник.

1. **Вводной автоматический выключатель (2P)** — рассматриваем как
   двухполюсный, размыкает L и N одновременно при выключении тумблера или
   срабатывании защиты. Номиналы на выбор: 16/25/32/40/63 А. Характеристика C
   (стандартная для бытового ввода).
2. **УЗО (2P, без защиты от сверхтоков)** — реагирует только на ток утечки.
   Номинал тока утечки 30 мА (для розеточных групп) или 100 мА (для ввода/
   противопожарный). Не защищает от перегрузки и КЗ — это важная обучающая
   деталь, частая ошибка новичков ("УЗО вместо автомата").
3. **Дифференциальный автомат (АВДТ, 2P)** — совмещает функции автомата
   (перегрузка, КЗ) и УЗО (утечка). Указываем номинал тока и тока утечки.
4. **Автоматический выключатель отходящей линии (1P или 2P)** — номиналы
   6/10/16/20/25 А, характеристика B или C.
5. **Реле напряжения (УЗМ-подобное, 1P по факту, но с транзитным N)** —
   входная клемма L и N, выходная клемма L (N идёт транзитом, не
   коммутируется). Настраиваемые пороги Umin и Umax. При выходе напряжения
   сети за пределы — размыкает выходную фазу с задержкой (см. 2.5).
6. **Трёхпозиционный переключатель "Сеть — 0 — Генератор" (ввод-резерв,
   2P)** — механический переключатель с тремя положениями: подключить
   нагрузку к сети, отключить всё (положение "0"), подключить нагрузку к
   резервному источнику (генератору). Главное обучающее свойство — он
   физически исключает одновременное подключение двух источников
   (предотвращает встречное включение генератора в сеть).
7. **Шина DIN (контактная шина для распределения L между автоматами)** —
   пассивный элемент, просто электрически объединяет несколько точек.
8. **Нулевая шина (N) и шина заземления (PE)** — пассивные шины, к которым
   подключаются все N- и PE-проводники потребителей. PE шина обязательно
   имеет соединение с контуром заземления (внешний узел, который можно
   "оборвать" для демонстрации, что произойдёт при отсутствии заземления).
9. **Нагрузка / индикаторная лампа** — простой потребитель с клеммами L, N
   (и опционально PE для корпуса). Используется как индикатор: горит, если
   на неё подано рабочее напряжение L-N в допустимых пределах; не горит,
   если цепь разорвана; мигает / гаснет аварийно при перенапряжении (в
   модели — просто визуальный индикатор "повреждена" при длительной работе
   на напряжении выше допустимого, см. 2.5).
10. **Источник: ввод от сети + опциональный второй источник "генератор"**
    — у генератора те же параметры (напряжение, возможность выключить), это
    нужно для трёхпозиционного переключателя.

Полностью out-of-scope для MVP (можно упомянуть как "следующие шаги" в
README, но не реализовывать): контакторы с катушками управления и логикой
самоблокировки, трёхфазная сеть, счётчики электроэнергии, УЗИП
(ограничители перенапряжения), таймеры/фотореле, частотные преобразователи.

### 2.4 Правила соединений (что можно и нельзя соединять)

Эти правила — основа "проверки ошибок", о которой просил пользователь.
Каждое правило должно приводить к понятному сообщению на языке пользователя,
а не к технической ошибке.

**Структурные правила (проверяются всегда, независимо от того, подано
питание или нет):**

- Клемма L можно соединять только с клеммой L (или с универсальным входом
  шины L). Соединение L↔N — это имитация короткого замыкания фаза-ноль и
  должна обрабатываться как авария (см. 2.5), а не как структурная ошибка,
  потому что в реальности именно так и происходит КЗ — провода замыкаются
  физически.
- Нельзя подключить два разных источника питания (сеть и генератор) к одной
  и той же точке цепи одновременно, **кроме как через трёхпозиционный
  переключатель**, который физически это исключает. Если пользователь
  пытается соединить выход "сеть" и выход "генератор" проводом напрямую
  (без переключателя между ними) — это структурная ошибка: "Нельзя
  напрямую соединять два источника питания — это вызовет опасное встречное
  включение. Используйте переключатель Сеть-0-Генератор."
- К одной клемме можно подключить более одного провода только если это явно
  предусмотрено (шины); для клемм компонентов (вход/выход автомата и т.п.) —
  максимум один провод на клемму. Попытка завести второй провод даёт
  сообщение "К этой клемме уже подключен провод. Используйте шину, если
  нужно разветвление."
- Нельзя подключить выход аппарата к его же входу (короткая петля сама на
  себя) — предупреждение "Это замыкает выход аппарата на собственный вход".

**Логические правила, которые не блокируют соединение, а влияют на
результат симуляции (это и есть "обучение через ошибку"):**

- **УЗО/АВДТ без целостного N** — если через УЗО не проходит и L, и N
  (например, N подключен мимо УЗО, прямо с шины), УЗО физически может быть
  собрано, но при подаче питания должно появиться предупреждение: "Ноль не
  проходит через УЗО — устройство не сможет корректно отслеживать ток
  утечки. УЗО должно коммутировать оба провода: и фазу, и ноль." В реальной
  схеме это типичная ошибка монтажа.
- **Автомат после УЗО с номиналом выше, чем у УЗО** — по правилам выбора
  защит автомат, защищающий конкретную линию, должен быть на ступень меньше
  номинала УЗО, иначе при перегрузке сгорит само УЗО, а не сработает
  автомат. Если пользователь ставит, например, УЗО 25А и за ним автомат 32А
  — предупреждение: "Номинал автомата (32А) больше номинала УЗО (25А).
  При перегрузке может выйти из строя УЗО, а не автомат. Автомат должен
  быть на ступень ниже."
- **PE подключен к N где-либо после УЗО (повторное соединение N-PE)** —
  это нарушение, которое делает УЗО неработоспособным или вызывает ложные
  срабатывания (потому что часть тока возвращается по PE, минуя
  дифференциальный трансформатор). Сообщение: "Соединение N и PE после УЗО
  нарушает работу защиты — часть обратного тока идёт мимо УЗО."
- **Нагрузка подключена без PE при наличии корпуса** — не блокирует работу,
  но даёт предупреждение про риск поражения током при пробое изоляции на
  корпус (для обучающего контекста — это важно показать).
- **Реле напряжения с обрывом N (транзитного)** — если транзитный N реле
  физически не подключен (хотя по паспорту это не обязательно, но если
  пользователь оставил клемму N висящей при том, что нагрузка требует N) —
  предупреждение про разрыв цепи нагрузки, а не про сам прибор.

### 2.5 Логика срабатывания защит (упрощённая, но физически осмысленная)

Это ядро движка симуляции. Все величины и кратности взяты из реальных
характеристик, чтобы цифры на экране были правдоподобными, но финальная
модель — дискретная (по тикам), а не точная интегральная.

**Автоматический выключатель (характеристики B/C/D):**

Кратность тока к номиналу `k = I_load / I_rated`.

- `k <= 1.0` — нормальная работа, бесконечно долго.
- `1.0 < k < 1.45` — обычно не приводит к отключению (зона
  неопределённости теплового расцепителя); в модели — не отключаем.
- `1.45 <= k < magnetic_threshold` — тепловой расцепитель, время
  срабатывания убывает от единиц секунд до ~1 часа в зависимости от
  перегрузки. Для простоты в модели используем грубую таблицу:
  - `1.45–2.0` → отключение через ~30–60 сек (имитация "тепловой защиты")
  - `2.0–3.0` → отключение через ~5–10 сек
- `k >= magnetic_threshold` — электромагнитный (мгновенный) расцепитель,
  отключение почти мгновенно (доли секунды). `magnetic_threshold` зависит
  от характеристики:
  - **B**: kратность 3–5 (берём нижнюю границу 3 как порог мгновенного
    срабатывания для модели)
  - **C**: кратность 5–10 (порог 5)
  - **D**: кратность 10–20 (порог 10)

`trip_reason` при тепловом — `"overload"`, при электромагнитном —
`"short_circuit"`.

**УЗО (без защиты от сверхтоков) и дифференциальный автомат (с защитой):**

- Дифференциальный ток (ток утечки) моделируем не физическим расчётом, а
  явным пользовательским действием — кнопка "Сымитировать утечку тока" у
  конкретной нагрузки/линии, плюс параметр "величина утечки" (мА),
  опционально можно задать по умолчанию фиксированное значение,
  превышающее номинал УЗО (например, всегда симулировать 50 мА при тесте,
  что больше типичных 30 мА).
- Если включена утечка и она `>= rated_leak_mA` на линии, защищённой этим
  УЗО/АВДТ → отключение почти мгновенно (на практике до 0.3 с, в модели —
  "почти сразу", `trip_reason: "leak"`.
- Если утечка `< 0.5 * rated_leak_mA` → НЕ отключается (имитация
  неотключающего тока).
- УЗО **не реагирует на перегрузку и КЗ** — если ток превышает номинал
  коммутации УЗО (обычно равен или на ступень выше защищающего автомата),
  оно физически может перегреться, но в модели для простоты: если нет
  отдельного автомата перед/после УЗО и ток превышает паспортный предел
  УЗО — выводим предупреждение "УЗО перегружено, требуется автомат защиты
  от перегрузки", но визуально можно либо считать, что оно "сгорает"
  (дальше неработоспособно до замены), либо просто держать предупреждение
  без отключения — решение оставляю на этапе реализации UI, но **дифавтомат
  должен отключаться и по перегрузке** (по логике автомата выше) — это и
  есть его отличие от простого УЗО, и именно это нужно показать наглядно.

**Реле напряжения (УЗМ-подобное):**

Параметры по умолчанию (взяты из реальных приборов УЗМ-51М, см. источники):
- Регулируемый нижний порог `Umin`: диапазон настройки 100–210 В
  (по умолчанию ставим 170–190 В как рекомендуемое значение).
- Регулируемый верхний порог `Umax`: диапазон настройки 240–290 В
  (по умолчанию 250–255 В).
- Если текущее напряжение сети `U` выходит за `[Umin, Umax]`:
  - Превышение верхнего порога — отключение быстрое, задержка порядка
    0.2 с (в модели — "почти мгновенно", `trip_reason: "overvoltage"`).
  - Опционально, экстремальное превышение (выше ~300 В) — мгновенное,
    задержка ~20 мс.
  - Понижение ниже нижнего порога — отключение с задержкой порядка 10 с
    (в модели — таймер на 5–10 секунд, чтобы пользователь видел задержку
    и понимал, что реле "ждёт", прежде чем отключить, в отличие от
    мгновенного отключения по перенапряжению).
  - Экстремальное понижение (ниже ~80 В) — более быстрое отключение,
    порядка 0.5 с.
- Реле коммутирует **только фазу**; N идёт транзитом и не размыкается —
  это важная обучающая деталь, обязательно отразить в схеме клемм
  компонента (вход/выход L отдельно, N — просквозная клемма без реле).
- После возврата напряжения в допустимый диапазон — автоматическое
  повторное включение (АПВ) с задержкой (в модели 5–10 секунд), визуально
  показываем обратный отсчёт.
- У реле два индикатора как у реальных приборов: "норма/авария"
  (зелёный/красный) и "реле включено" (жёлтый) — желательно сохранить в UI
  как минимум двухцветную индикацию состояния.

**Обрыв нуля (отдельный тестовый сценарий):**

- Если пользователь нажимает кнопку "оборвать N" на магистрали до щита —
  на однофазных потребителях, в зависимости от того, через что они
  подключены:
  - Если нагрузка подключена через реле напряжения с честным транзитом N —
    при отсутствии N само реле должно показывать аварийный режим (нет
    напряжения относительно N → измеряемое напряжение = 0 или
    неопределено) и отключать нагрузку — это реальное защитное свойство
    реле напряжения, его стоит подчеркнуть как "почему реле напряжения
    дополнительно защищает от обрыва нуля".
  - Если нагрузки подключены без реле напряжения — индикатор лампы должен
    либо не светиться (полный обрыв N до щита — нет замкнутой цепи вообще),
    либо в более продвинутом сценарии (мигание/перекос фаз при нескольких
    нагрузках с разным сопротивлением) — это можно оставить как
    необязательное расширение, помечаем как "не входит в MVP".

**Трёхпозиционный переключатель Сеть-0-Генератор:**

- Три дискретных положения: `"network"`, `"off"`, `"generator"`.
- В положении `"network"` — выход подключен только к клеммам сети, вход от
  генератора электрически изолирован (внутри модели — не передаём
  состояние от генератора на выход).
- В положении `"off"` — выход не подключен ни к одному источнику.
- В положении `"generator"` — выход подключен только к клеммам генератора,
  сеть изолирована.
- Это единственный компонент, которому разрешено иметь два независимых
  входа от двух разных источников — потому что он гарантирует, что они
  никогда не замкнуты одновременно. Это и есть обучающий смысл компонента:
  объяснить, почему нельзя просто параллельно врезать генератор в щит без
  такого переключателя (опасность подачи напряжения обратно в сеть).

### 2.6 Алгоритм симуляции (как это считать технически)

Рекомендуемый подход — **дискретная симуляция по тикам** (например, раз в
100–250 мс реального времени, с ускоренным "виртуальным" временем для
задержек срабатывания — см. ниже про игровое время vs реальное), а не
непрерывное аналоговое моделирование:

1. Построить граф из клемм и проводов. Для каждого проводника типа L —
   найти все связанные компоненты через замкнутые (включенные,
   несработавшие) коммутирующие элементы, начиная от каждого источника
   питания.
2. Для каждой нагрузки определить: подключена ли она к L и к N от одного и
   того же логического источника (сеть либо генератор, не вперемешку);
   если да — вычислить эффективное напряжение на нагрузке (с учётом текущего
   значения "напряжение сети" или "напряжение генератора", тестовых
   слайдеров, и состояния реле напряжения на пути).
3. Просчитать ток через каждый коммутирующий аппарат как сумму
   номинальных токов включенных и исправно работающих нагрузок "ниже по
   течению" от него (для MVP: у каждой нагрузки/лампы есть фиксированный
   номинальный ток, который пользователь либо задаёт, либо берёт из
   пресета "лампа = 0.5А", "обогреватель = 10А", "стиральная машина = 6А" и
   т.п. — набор пресетов нагрузок стоит сделать небольшим и понятным).
4. Применить правила срабатывания (2.5) к каждому аппарату защиты на
   основе посчитанного тока/утечки/напряжения. Если аппарат должен
   сработать — выставить `tripped: true`, `trip_reason`, и пересчитать всю
   схему заново (отключенный аппарат разрывает цепь ниже себя).
5. Обновить визуальное состояние каждого компонента и каждой нагрузки
   (индикаторы горят/не горят/мигают, аппараты показывают
   включено/выключено/авария).

**Про время:** задержки в реальных приборах — от десятков миллисекунд до
10 секунд (и 6 минут для повторного включения реле напряжения). Для UX
приложения секунды ожидания — это нормально и даже полезно для обучения
(пользователь видит, что отключение по перегреву не мгновенное, в отличие
от КЗ), а 6 минут реального ожидания — нет. Рекомендация: использовать
реальные секунды для коротких задержек (0.2–10 с) и **визуально
ускоренный таймер с явной подписью "имитация ожидания: 6 мин → 6 сек"**
для длинных, чтобы не врать пользователю о реальных цифрах, но и не
заставлять его ждать в браузере 6 минут.

### 2.7 Сообщения об ошибках и предупреждения — общий принцип

Каждое правило из 2.4 и 2.5 должно генерировать структурированный объект:

```js
{
  severity: "error" | "warning" | "info",
  code: "RULE_ID",
  message_short: "для бейджа на компоненте",
  message_full: "для панели объяснений — что произошло и почему, со ссылкой на физику",
  related_components: ["id1", "id2"]
}
```

`error` — структурно невозможное соединение (блокируется до создания, или
явно помечается красным и схема не запускается, пока не исправлено).
`warning` — соединение возможно, схема "работает", но это плохая практика
или опасность (УЗО без N, нет PE, неправильная ступенчатость номиналов).
`info` — поясняющая подсказка, не ошибка (например, "автомат отключился из-
за перегрузки — подождите остывания биметаллической пластины перед
повторным включением", чтобы научить, что мгновенно включать обратно
сработавший от перегрузки автомат — плохая идея).

## 3. План реализации (для будущей сессии, когда придём писать код)

### 3.1 Архитектура

Разделить на три слоя, чтобы дизайн от Claude Design можно было подключить
без переписывания логики:

1. **`model/`** — чистые данные и правила без DOM: определения типов
   компонентов (каталог из раздела 2.3), граф схемы, функции проверки
   правил (2.4), функция одного тика симуляции (2.6), генератор объектов
   ошибок/предупреждений (2.7). Это чистый JS/TS, тестируемый без браузера.
2. **`engine/`** — обёртка над `model/`, которая управляет игровым
   временем (тики, таймеры срабатывания, ускоренные длинные задержки),
   хранит текущее состояние схемы (компоненты, провода, источники), и
   предоставляет реактивный API (например, простой паттерн
   subscribe/notify или React state) для UI-слоя.
3. **`ui/`** — визуальный слой: DIN-рейка, перетаскивание компонентов,
   рисование проводов, индикаторы, панель ошибок, слайдеры тестовых
   воздействий (напряжение сети, утечка, обрыв N). Этот слой — то место,
   куда интегрируется дизайн от Claude Design. UI слой обращается к
   `engine/` только через его публичный API и не содержит электротехнической
   логики напрямую.

Рекомендация по технологиям: одиночный React-компонент (или несколько
файлов, собранных в один HTML/JS артефакт) с локальным state, без внешнего
бэкенда. Все вычисления — на клиенте. Это соответствует ограничениям
окружения Claude.ai (артефакты не могут делать сетевые вызовы, кроме
заданных CDN-библиотек).

### 3.2 Этапы реализации

Подход — **design-first**: сначала собираем визуальный костяк по готовому
макету Claude Design, затем наращиваем логику. Граница `model/engine/ui`
закладывается с первого этапа (model/engine — заглушки с типами), чтобы при
наращивании симуляции не переписывать UI.

**Этап 1 — Визуальный костяк приложения.**
- Скаффолд проекта (React + Vite, single-page, без бэкенда).
- Структура папок `model/` `engine/` `ui/` с заглушками типов и пустыми
  функциями в первых двух — чтобы UI с самого начала ходил через тот же
  API, который потом реализуем.
- Перенести дизайн-токены из выбранного направления Claude Design (цвета,
  шрифты Space Grotesk / Inter / JetBrains Mono, отступы, радиусы).
- Сверстать общий шелл: шапка проекта, левая палитра компонентов,
  центральная рабочая область с пустой DIN-рейкой и шинами L/N/PE,
  панель «лог событий», нижняя «тестовая панель» (слайдер напряжения,
  кнопки утечки/обрыва N/КЗ — пока без действий).
- Статичные модули в палитре по образу `Module.dc.html`: каждый из 9 типов
  компонентов из 2.3 нарисован как узнаваемый DIN-модуль.
- 4 статичных макета состояний (норма / срабатывание УЗО / ошибка сборки /
  перенапряжение) как переключаемые скрины — чисто визуально, без логики,
  для проверки, что дизайн покрывает все сценарии из 2.5.
- Адаптив, доступность (состояния различимы не только цветом — иконки/
  текстовая подпись дублируют цвет, см. 3.3).

**Этап 2 — Drag-and-drop на DIN-рейку (без симуляции).**
- Перетаскивание модулей из палитры в слоты рейки.
- Сохранение раскладки в state модели (`model/scheme.ts`).
- Удаление, перемещение, выбор модуля → показ его параметров в инспекторе.
- Никакой валидации соединений и токов — просто расстановка.

**Этап 3 — Соединение проводами + структурные правила (2.4).**
- Клик-клик или drag по клеммам создаёт провод.
- Цветовая маркировка проводов по ПУЭ/МЭК (L — коричневый/красный, N —
  синий, PE — жёлто-зелёный).
- Структурные ошибки из 2.4 (L↔L, два источника напрямую, один провод на
  клемму, петля сам-на-себя) — блокировка/подсветка в момент создания.
- Логические предупреждения 2.4 — на следующем этапе, когда появится
  модель тока.

**Этап 4 — Модель и движок симуляции (2.5, 2.6).**
- Реализовать `model/` целиком: каталог компонентов, граф, функция тика,
  генератор сообщений 2.7.
- `engine/` — игровое время, таймеры срабатывания, ускоренные длинные
  задержки, реактивный API.
- Кнопка «Подать питание» и слайдеры тестовой панели подключаются к
  источнику.
- Индикаторы и состояния модулей реактивно обновляются из движка.
- Unit-тесты на ручные сценарии из ТЗ («ввод → шина → автомат 16А →
  лампа» горит; «то же с обрывом N» не горит).

**Этап 5 — Панель объяснений и обучающий режим.**
- Структурированный лог сообщений из 2.7 с фильтром по severity.
- Постоянно видимое «почему сработало» рядом с индикатором аварии.
- Подсказки `info` (например, «дай автомату остыть»).
- Опционально — пошаговые задания, помечаем как «после MVP».

**Этап 6 — Полировка.**
- Сохранение/загрузка сборки в localStorage.
- Анимации срабатывания, обратный отсчёт АПВ реле напряжения.
- Финальный QA по сценариям из 2.5.

### 3.3 Что важно сохранить при любой реализации дизайна

Эти требования должны быть переданы и в промпт для Claude Design (см.
отдельный файл), потому что они одновременно и про логику, и про
визуальное представление:

- Цветовая маркировка проводов по ПУЭ/МЭК (L/N/PE) должна быть видна и
  узнаваема — это часть обучающей ценности приложения, а не просто стиль.
- Состояния компонентов (включен / выключен / авария / предупреждение)
  должны быть различимы не только по цвету (для доступности) — иконка или
  текстовая подпись дублирует цвет.
- Должно быть постоянно видимое место для "почему сработало" — пользователь
  не должен гадать, что значит мигающий красный индикатор; рядом всегда
  есть текстовое объяснение.
- DIN-рейка и компоненты должны быть узнаваемо похожи на реальные модульные
  электротехнические изделия (прямоугольные корпуса 17.5/35 мм, рычажки
  включения, окошки индикации) — это не абстрактная схема-конструктор, а
  стилизованная, но узнаваемая имитация реального щита, что помогает
  переносу знаний в реальный монтаж.
