# План: лесенка проводов (wire lanes)

## Контекст для нового агента

Пользователь скинул фото реального щита и хочет, чтобы провода в нашем
приложении формировались так же: каждый провод идёт по **своей
горизонтальной дорожке (lane)**, выходы из клемм — строго вертикально,
без общего канала. Сейчас все провода одного проводника (L/N/PE) делят
один X-канал и одну Y-полосу `midY` — поэтому 5 проводов от шины к 5
автоматам ложатся друг поверх друга.

Фото-референс: на фото слева вводной 2P C50, реле напряжения,
5 отходящих автоматов (C10, C16×3, C25). От верхней шины N к каждому
автомату — лесенка из 5 синих проводов на разных Y. Снизу от 5 нагрузок
к 5 автоматам — такая же лесенка, выходы строго вертикально из клемм.

Ответы пользователя на уточняющие вопросы из прошлой сессии:
1. **Lane assignment** — авто по дистанции до цели (ближний провод →
   дорожка рядом со стартом, дальний → дальше).
2. **Обход модулей** — провода идут МЕЖДУ модулями (есть зазор),
   не поверх. Это и косметика (реалистичность), и обоснование, почему
   дорожки не пересекают корпуса.
3. **Вертикальные отступы** — строго по Y, X-смещение из клеммы НЕ
   делаем. Клеммы и так на разных X.
4. **Обратная совместимость** — новые провода рисуются по новой логике,
   уже размещённые остаются как есть (lane хранится в Wire).

Открытый вопрос, который я задал в конце прошлой сессии и на который
пользователь ещё не ответил:

> На фото N идёт через верхнюю шину сверху, и лесенка формируется в
> пространстве между шиной и автоматами. У нас шины N и L расположены
> ниже DIN-рейки. Планирую различать «полосу» (zone) автоматически по
> Y-координатам endpoint'ов: оба endpoint выше рейки → верхняя полоса,
> оба ниже → нижняя, через рейку → отдельная зона. Уточнить у
> пользователя в начале новой сессии, ок ли это правило.

## Что меняем

### Модель

1. **`src/model/types.ts` (или там, где определён `Wire`)** — добавить
   optional поле `lane?: number`. У старых проводов поле отсутствует —
   они рисуются по-старому (`manhattanPath`).

2. **`src/model/scheme.ts`, reducer `add_wire`** — при добавлении
   нового провода вычислять lane:
   - определить «полосу» (zone) для пары (from, to) — см. шаг 3 ниже;
   - собрать все существующие провода в той же полосе;
   - отсортировать новый провод среди них по горизонтальной дистанции
     `|from.x − to.x|`;
   - присвоить индекс позиции в сортировке.
   Ближний → lane 0 (дорожка рядом со стартом), дальний → lane N.

### Геометрия

3. **`src/model/layout.ts`** — добавить:
   - `laneStepRem` — шаг между соседними дорожками (≈ 0.35rem);
   - `moduleGapRem` — визуальный зазор между корпусами DIN-модулей
     (≈ 0.15rem), косметика для реалистичности;
   - функция `wireZone(from, to, layout)` → строковый идентификатор
     полосы (например, `"above_rail_1"`, `"below_rail_1"`,
     `"between_rails"`);
   - функция `lanePathY(zone, lane, layout)` → Y-координата
     горизонтали для данной зоны и индекса дорожки.

### Рендер

4. **`src/ui/Workspace.tsx`, `WirePath`** — если `wire.lane !==
   undefined`, использовать новый алгоритм:
   ```
   manhattanPathLane(from, to, laneY) →
     [from, {x: from.x, y: laneY}, {x: to.x, y: laneY}, to]
   ```
   Без обхода препятствий, без общего канала. Если `wire.lane`
   undefined — старый `manhattanPath` (для legacy-проводов).

### Persistence

5. **Ничего не делаем.** `Wire` сериализуется целиком в
   `src/model/persistence.ts`, lane сам попадёт в localStorage / JSON.

## Success criteria

- [ ] Сценарий: кладу шину L сверху + 5 автоматов снизу, соединяю
      шину с каждым автоматом — получаю лесенку из 5 разных Y,
      провода не накладываются. Вертикали выходят строго из X клемм.
- [ ] Удаляю средний провод — оставшиеся 4 НЕ перестраиваются (lane
      зафиксирован в каждом Wire).
- [ ] Добавляю новый провод в ту же полосу — он получает следующий
      свободный lane.
- [ ] Перезагружаю страницу — лесенка сохраняется (через
      persistence).
- [ ] Существующие сохранённые схемы из localStorage загружаются и
      рисуются по-старому (lane undefined → старый manhattanPath).
- [ ] `npm run build && npm run lint && npm run test` — без ошибок.
- [ ] Визуальная проверка: dev-server, собрать схему как на фото
      (вводной 2P + реле + 5 отходящих), проводка похожа на
      референс.

## Что НЕ делаю (out of scope этого изменения)

- Не перерисовываю существующие провода (пользователь сказал
  «только новые»).
- Не делаю двойную лесенку по X (выходы из клемм только по Y).
- Не удаляю `manhattanPath` / `safeYBands` / `findClearColumn` —
  нужны для legacy-проводов.
- Не делаю UI для ручного выбора lane в инспекторе.
- Не трогаю расхождения с ТЗ из §5 CLAUDE.md (2P отходящий, warning
  для реле без N, warning «голое УЗО») — отдельные задачи.
